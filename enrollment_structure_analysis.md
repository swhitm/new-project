# Structural Analysis of Enrollment Manager Project

Based on the method map and the code in `enrollment_starter.py`, this analysis identifies issues with mixed responsibilities across service and database layers, instances where database code makes service-level decisions, cross-layer methods, and use of global state. The code is currently procedural, with functions handling both service orchestration (e.g., business logic like enrollment rules) and database operations (e.g., SQL queries), while relying heavily on global constants for state. This leads to tight coupling, where changes in one area can ripple across the system.

## Key Findings
- **Mixed Responsibilities**: Several functions marked as "Service" (e.g., `enroll_with_key`, `soft_unenroll_student`) directly execute SQL operations (INSERT/UPDATE), which should be abstracted to a database layer. Conversely, functions marked as "Database" (e.g., `get_available_course_keys`) use service-level utilities like `connect()` and global paths, blurring the lines. This violates the single responsibility principle, as service functions are handling data persistence logic.
- **Database Code Making Service Decisions**: While not explicitly making business decisions, database functions like `get_student_enrollments` hardcode filters (e.g., `status = STATUS_ENROLLED`), which embeds service-level logic (what constitutes "enrolled") into data access. This makes it hard to change enrollment rules without altering queries.
- **Cross-Layer Methods**: Functions like `export_database_snapshot` (Service) perform both data retrieval (calling database functions) and export logic (JSON writing), spanning data access and presentation layers.
- **Use of Global State**: Constants like `DB_PATH`, `CURRENT_STUDENT`, and `AVAILABLE_COURSE_KEYS` are accessed globally, making functions dependent on external state. This affects reproducibility and testing, as functions aren't self-contained.

These issues hurt **scalability** by making it difficult to introduce changes like switching databases, adding caching, or scaling to multiple users—everything is intertwined, so modifications require widespread refactoring. They hurt **maintainability** by increasing bug risk (e.g., global state changes can break unrelated functions), reducing testability (hard to isolate units), and complicating collaboration (developers must understand the entire codebase to change one part).
